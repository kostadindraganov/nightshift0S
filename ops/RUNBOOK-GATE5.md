## 1. nftables egress — activate uid-scoped filtering for the agent user

### Why this is needed

`src/egress/guard.ts:assertEgressOrRefuse` blocks unattended runs on untrusted repos
until `egressActive()` detects the table `inet nightshift_egress_uid<uid>` via
`nft list tables`. Without it `sandbox.unattendedUntrustedRepos` must be manually forced
to `true` (as is currently the case in `nightshift.config.json`). With it, the scheduler
unblocks automatically for untrusted repos under cron / webhook triggers.

`src/egress/allowlist.ts` generates a default-DROP ruleset scoped to one UID; the
allowed hosts are `github.com`, `api.github.com`, `api.anthropic.com`, `api.openai.com`
plus any extras from `NIGHTSHIFT_EGRESS_HOSTS`. The `nft(8)` ruleset lives only in
kernel memory — it must be re-applied after every reboot.

The dedicated agent user `nightshift-agent` (uid 999, gid 991) already exists on the
host (`id nightshift-agent` → `uid=999(nightshift-agent) gid=991(nightshift-agent)`).
Agent processes spawned by bwrap-lite run under this uid (`--uid 999`), set via
`NIGHTSHIFT_AGENT_UID=999` in `main.ts` / `/etc/nightshift/env`.

### Prerequisite — deploy the agent-uid change

`src/runs/spawn.ts:resolveAgentIds` reads `NIGHTSHIFT_AGENT_UID` / `NIGHTSHIFT_AGENT_GID`
from env. The service must be running with these set so that every bwrap spawn uses uid
999 (not the service uid 1000). Confirm first:

```bash
sudo systemctl cat nightshift | grep -E 'AGENT_UID|EnvironmentFile'
sudo cat /etc/nightshift/env | grep -E 'AGENT_UID|AGENT_GID'
```

If the vars are absent, add them to `/etc/nightshift/env`:

```bash
sudo tee -a /etc/nightshift/env <<'EOF'
NIGHTSHIFT_AGENT_UID=999
NIGHTSHIFT_AGENT_GID=991
EOF
sudo systemctl restart nightshift
```

Confirm the service started cleanly:

```bash
sudo journalctl -u nightshift -n 20 --no-pager
curl -s http://127.0.0.1:3000/healthz
```

### Apply the egress ruleset

After the agent-uid change is live, apply the nftables ruleset as root. The ruleset
is generated by `src/egress/applyCli.ts` and loaded with `nft -f`:

```bash
sudo NIGHTSHIFT_EGRESS_UID=999 bash /home/nightshift/work/ops/egress-apply.sh
```

To add extra hosts beyond the default allowlist (e.g. a self-hosted Forgejo):

```bash
sudo NIGHTSHIFT_EGRESS_UID=999 \
     NIGHTSHIFT_EGRESS_HOSTS="codeberg.org api.groq.com" \
     bash /home/nightshift/work/ops/egress-apply.sh
```

Confirm the table is loaded:

```bash
sudo nft list tables
# expect: table inet nightshift_egress_uid999

sudo nft list table inet nightshift_egress_uid999
# expect: default policy drop chain + allowlist set
```

### Make rules survive reboot (optional but recommended)

The ruleset lives only in kernel memory. To reload it automatically, add an
`ExecStartPre=+` line to `ops/nightshift.service` (the `+` prefix gives that step
root, while `User=nightshift` applies only to `ExecStart`):

```ini
[Service]
# ... existing lines ...
ExecStartPre=+/usr/bin/bash -c 'NIGHTSHIFT_EGRESS_UID=999 bash /opt/nightshift/ops/egress-apply.sh || true'
ExecStart=/home/nightshift/.bun/bin/bun run /opt/nightshift/src/server/main.ts
```

After editing the unit file:

```bash
sudo cp /home/nightshift/work/ops/nightshift.service /etc/systemd/system/nightshift.service
sudo systemctl daemon-reload
sudo systemctl restart nightshift
```

Alternatively, install a separate `nightshift-egress.service` that runs before
`nightshift.service` using `Before=` / `Requires=` ordering — this keeps the unit
file clean.

### Observable result

Once the table is present, `src/egress/guard.ts:egressActive()` returns `true`. The
unattended scheduler then permits runs on untrusted repos without requiring
`sandbox.unattendedUntrustedRepos=true` in config. You may remove or set that flag to
`false` in `nightshift.config.json` for the stricter production posture:

```json
{
  "sandbox": {
    "unattendedUntrustedRepos": false
  }
}
```

Restart the service after changing the config file.

To tear down the ruleset (e.g. for maintenance):

```bash
sudo NIGHTSHIFT_EGRESS_UID=999 bash /home/nightshift/work/ops/egress-teardown.sh
```

## 2. CMA provider (Anthropic Managed Agents) — 7.5

### Why this is needed

`src/providers/cma.ts` implements the CMA `ProviderDriver` (kind `"api"`). It is
registered in the driver registry behind `providers.cmaEnabled` (default `false`). The
driver sends requests to `https://api.anthropic.com/v1/messages` (or `CMA_BASE_URL`)
with `anthropic-beta: managed-agents-2025-01-01` and `x-api-key: <CMA_API_KEY>`. The
session continuation endpoint is `/sessions/<id>/messages` — **these wire shapes are
speculative and may need adjustment once the live CMA endpoint is accessible**; the
conformance probe will reveal any mismatch.

### Set secrets and enable the driver

Add to `/etc/nightshift/env`:

```bash
sudo tee -a /etc/nightshift/env <<'EOF'

# 7.5 CMA — Anthropic Managed Agents provider
CMA_API_KEY=sk-ant-...          # required: your Anthropic API key
CMA_MODEL=claude-opus-4-5       # required: model id
# CMA_BASE_URL=https://api.anthropic.com/v1  # optional override
EOF
```

Enable the driver in `nightshift.config.json` (at `/opt/nightshift/nightshift.config.json`):

```json
{
  "providers": {
    "cmaEnabled": true,
    "cmaModel": "claude-opus-4-5"
  }
}
```

(`cmaModel` in config is the fallback when `CMA_MODEL` env is absent; the env var wins
per `src/providers/cma.ts:231`.)

Restart the service:

```bash
sudo systemctl restart nightshift
```

### Run the conformance probe

The conformance harness (`src/providers/conformance.ts`) proves or fails the declared
capabilities (`resume`, `structured_output`, `cost_reporting`) and writes the results
to the `providers` DB table. Trigger it via the API:

```bash
curl -s -X POST http://127.0.0.1:3000/providers/conformance \
  -H "Authorization: Bearer $(grep NIGHTSHIFT_API_TOKEN /etc/nightshift/env | cut -d= -f2)" \
  -H "Content-Type: application/json" \
  -d '{"provider": "cma"}'
```

Inspect the result:

```bash
curl -s http://127.0.0.1:3000/providers/health \
  -H "Authorization: Bearer $(grep NIGHTSHIFT_API_TOKEN /etc/nightshift/env | cut -d= -f2)" \
  | jq '.[] | select(.name == "cma")'
```

A `status: "healthy"` response with `proven.resume=true` and
`proven.structured_output=true` means the driver is fully operational and the router
may assign it coder/reviewer/planner roles. A `status: "unproven"` or failed
`resume` probe indicates the live endpoint wire shape differs from what `cma.ts`
expects — inspect the error field and update `runOnce`/`resumeOnce` in
`src/providers/cma.ts` accordingly, then re-run the probe.

## 3. Preview environments — 7.4

### Why this is needed

`src/preview/preview.ts:CommandDeployer` runs operator-provided shell commands to
spin up and tear down per-run preview URLs (`run-<id>.<domain>`). The URL is allocated
by `allocateUrl(runId, domain)` as soon as the run starts. The deployer receives
`[<deployCommand...>, runId, url]` as positional args (no shell interpolation). The
idle reaper (`startPreviewReaper`) tears down environments that have been silent for
`preview.idleReapMinutes` (default 30).

### Infrastructure prerequisites

1. **Wildcard DNS**: create a wildcard A/CNAME record `*.<domain>` pointing at the
   reverse proxy's public IP. Example (Cloudflare DNS or similar):
   ```
   *.preview.example.com  A  <proxy-ip>
   ```

2. **Reverse proxy** (Caddy, nginx, or Traefik) that routes `run-<id>.<domain>` to the
   correct backend. A minimal Caddy example:
   ```
   *.preview.example.com {
       tls {
           # Wildcard cert — requires DNS challenge provider
       }
       reverse_proxy 127.0.0.1:{env.PREVIEW_PORT_{args[0]}}
   }
   ```
   Your deploy script is responsible for registering each run's port/upstream with the
   proxy on deploy and removing it on teardown.

3. **Deploy and teardown scripts** — write two executable scripts on the Linux host.
   They receive positional args: `<runId> <url>` for deploy, `<runId>` for teardown.
   Minimal stubs to start:
   ```bash
   # /opt/nightshift/ops/preview-deploy.sh
   #!/usr/bin/env bash
   set -euo pipefail
   RUN_ID="$1"
   URL="$2"
   # Register $URL → upstream with proxy, start any per-run process, etc.
   echo "[preview-deploy] run=$RUN_ID url=$URL"

   # /opt/nightshift/ops/preview-teardown.sh
   #!/usr/bin/env bash
   set -euo pipefail
   RUN_ID="$1"
   # Deregister upstream, stop per-run process, etc.
   echo "[preview-teardown] run=$RUN_ID"
   ```
   ```bash
   chmod +x /opt/nightshift/ops/preview-deploy.sh
   chmod +x /opt/nightshift/ops/preview-teardown.sh
   ```

### Enable in config

In `/opt/nightshift/nightshift.config.json`:

```json
{
  "preview": {
    "enabled": true,
    "domain": "preview.example.com",
    "idleReapMinutes": 30,
    "deployCommand": ["/opt/nightshift/ops/preview-deploy.sh"],
    "teardownCommand": ["/opt/nightshift/ops/preview-teardown.sh"]
  }
}
```

(`deployCommand` and `teardownCommand` are read by `src/server/v2Boot.ts` through the
`PreviewConfig` type; they are not top-level `NightshiftConfig` fields but are read via
a type assertion from the raw config object — keep them nested under `preview`.)

Restart and verify:

```bash
sudo systemctl restart nightshift
curl -s http://127.0.0.1:3000/previews \
  -H "Authorization: Bearer $(grep NIGHTSHIFT_API_TOKEN /etc/nightshift/env | cut -d= -f2)"
# expect: [] (no previews yet; endpoint responds 200)
```

## 4. Multi-VM workers — 7.2

### Why this is needed

`src/scheduler/workerDaemon.ts` is a standalone daemon that runs on a second Linux host.
It POSTs to `POST /workers` (registration) and `POST /workers/:id/heartbeat` on an
interval. The control-plane `WorkerRegistry` (`src/scheduler/workers.ts`) tracks
liveness and reclaims stale workers after `workers.leaseSeconds`. The scheduler
consumes the registry to farm tasks to remote machines. `workers.enabled=true` on the
control plane is required for the registry endpoints to be active.

### Control-plane side

Enable in `/opt/nightshift/nightshift.config.json` on the primary host:

```json
{
  "workers": {
    "enabled": true,
    "heartbeatSeconds": 30,
    "leaseSeconds": 90
  }
}
```

Restart the service:

```bash
sudo systemctl restart nightshift
```

Verify the `/workers` endpoint is live:

```bash
curl -s http://127.0.0.1:3000/workers \
  -H "Authorization: Bearer $(grep NIGHTSHIFT_API_TOKEN /etc/nightshift/env | cut -d= -f2)"
# expect: {"workers": []}
```

### Second-machine side

On the second Linux host, install Bun and clone (or rsync) the nightshift repo:

```bash
curl -fsSL https://bun.sh/install | bash
# Clone or rsync the repo to /opt/nightshift-worker
git clone <repo-url> /opt/nightshift-worker
cd /opt/nightshift-worker && bun install
```

Set the required env vars (in `/etc/nightshift/env` or exported before the command):

```bash
export NIGHTSHIFT_CONTROL_PLANE_URL=http://<control-plane-ip>:3000
export NIGHTSHIFT_WORKER_TOKEN=<same NIGHTSHIFT_API_TOKEN as the control plane>
export NIGHTSHIFT_WORKER_ID=worker-2                 # unique across all workers
export NIGHTSHIFT_WORKER_HOST=worker-2.example.com   # human-readable label
export NIGHTSHIFT_WORKER_CAPACITY=2                  # concurrent task slots
```

Run the daemon:

```bash
bun run /opt/nightshift-worker/src/scheduler/workerDaemon.ts
```

The daemon is inert (exits 0) when `workers.enabled=false` in config or when any of
the three required vars (`NIGHTSHIFT_CONTROL_PLANE_URL`, `NIGHTSHIFT_WORKER_ID`,
`NIGHTSHIFT_WORKER_TOKEN`) is absent.

To run as a systemd service on the second machine, create
`/etc/systemd/system/nightshift-worker.service`:

```ini
[Unit]
Description=Nightshift remote worker daemon
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=nightshift
EnvironmentFile=/etc/nightshift/env
ExecStart=/home/nightshift/.bun/bin/bun run /opt/nightshift-worker/src/scheduler/workerDaemon.ts
Restart=on-failure
RestartSec=10
StandardOutput=journal
StandardError=journal

[Install]
WantedBy=multi-user.target
```

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now nightshift-worker
```

Confirm the worker registered on the control plane:

```bash
curl -s http://127.0.0.1:3000/workers \
  -H "Authorization: Bearer $(grep NIGHTSHIFT_API_TOKEN /etc/nightshift/env | cut -d= -f2)" \
  | jq '.workers'
# expect: [{id: "worker-2", host: "...", capacity: 2, ...}]
```

3. In V1, **the owner must click Merge** (auto-merge is off: `review.autoMergeEnabled=false`).

### Owner merge step (outward-facing — requires explicit go)

Push and merge are outward-facing actions against the real GitHub repo
`kostadindraganov/test`. They require explicit owner authorisation.

After the reviewer run completes (watch `journalctl -u nightshift -f` or the kanban
board — task 3 card will show state `approved`):

```bash
# Confirm the task is in state "approved" before merging
curl -s http://127.0.0.1:3000/tasks/3 \
  -H "Authorization: Bearer $(grep NIGHTSHIFT_API_TOKEN /etc/nightshift/env | cut -d= -f2)" \
  | jq '.state'
# expect: "approved"

# Merge PR #1 via gh CLI (requires GITHUB_TOKEN in env or gh auth)
gh pr merge 1 --repo kostadindraganov/test --merge --auto=false
```

Or merge via the GitHub web UI for PR #1 at:
`https://github.com/kostadindraganov/test/pull/1`

### What happens after merge

The forge service calls back `confirmMergeAndUnblock` in
`src/orchestrator/coder.ts`, which:

1. Transitions task 3: `merging → done` with `merge_sha` recorded.
2. Calls `recomputeReadiness` for all tasks in the same project — any task blocked on
   task 3 (by dependency) is promoted from `backlog → ready`.

This is the end-to-end confirmation that the full autonomous loop (schedule → code →
push → PR → review → merge → unblock) works on the live Linux host.

If the task is stuck in `review` and the reviewer run has failed, you can inspect the
run transcript:

```bash
curl -s http://127.0.0.1:3000/tasks/3/transcript \
  -H "Authorization: Bearer $(grep NIGHTSHIFT_API_TOKEN /etc/nightshift/env | cut -d= -f2)" \
  | jq '.events[-5:]'
```

And check journald for the reviewer one-shot error:

```bash

```
